# Delivery platform with real-time tracking and order workflow

A full delivery agency platform built on Express, Socket.IO, and SQLite. The backend implements JWT auth, a drivers CRUD, and an orders API with a multi-step status workflow (pending → assigned → picked_up → on_the_way → delivered). The frontend is two vanilla-JS SPAs: an admin dashboard (order management, live driver map, stats) and a driver app (order queue, GPS location sharing). Real-time communication pushes order events and driver positions to connected clients via Socket.IO rooms.

Watch for: hardcoded JWT secret with no production guard (confirmed), token stored in localStorage enabling XSS-to-token-theft escalation (confirmed), `GET /api/orders/:id` has no ownership check for drivers (confirmed), order code generation can collide under concurrency and crash with unhandled UNIQUE violation (likely), and both frontends fail to unwrap the `/api/auth/me` response object causing broken auth persistence on page refresh (confirmed).

## High-level view

Authentication uses bcrypt + JWT with a 12-hour TTL. The secret defaults to a static string and nothing in the code warns or refuses to start in production without a real secret. Both frontends store the token in localStorage and also receive an httpOnly cookie, but the Socket.IO and API calls exclusively use the localStorage copy, which is extractable by any XSS vector on the page.

The orders API is a factory function that closes over the Socket.IO instance. The workflow is enforced server-side with a transition map, which prevents out-of-order state changes. However the `GET /:id` route lets any authenticated user fetch any order by ID regardless of role, breaking the ownership boundary the `GET /` route establishes.

Socket.IO uses JWT auth middleware and role-based rooms. The disconnect handler always marks a driver offline immediately (no grace period), which creates false-offline flickering on brief network interruptions.

Both frontends assign `currentUser = await res.json()` from `/api/auth/me`, which returns `{ user: {...} }`. They then access `currentUser.name` and `currentUser.role` — both undefined. This means token-based auth persistence across page refresh is broken in both apps; only the initial login flow works.

1. **Hardcoded JWT secret** — `JWT_SECRET` defaults to `"dev-secret-change-me"` with no runtime check. Add an error or warning at startup when `JWT_SECRET` is not set via env var.
2. **Token in localStorage** — Both SPAs store the JWT in localStorage, making it extractable by any XSS. Rely on the httpOnly cookie for API calls and remove localStorage token storage.

4. **Order code collisions** — `generateCode()` uses `Date.now()` truncated to 4 chars plus 2 random chars. Under concurrent requests the timestamp portion can repeat, yielding duplicates that hit the UNIQUE constraint and return a 500. Use a retry loop or a sequence.
5. **Aggressive offline marking on disconnect** — Any socket disconnect (network glitch, tab backgrounding) immediately marks a driver offline and broadcasts `driver:offline`. Consider a grace period before marking offline.
6. **No rate limiting on login** — The login endpoint is brute-forceable without throttling.
7. **PUT /api/orders/:id cannot clear fields** — The update uses `|| null` as the COALESCE argument, so passing an empty string for a field doesn't clear it. Fields like `items`, `notes`, `customer_phone` cannot be unset once written.
8. **Broken auth persistence on page refresh** — Both frontends don't unwrap the `{ user: {...} }` envelope from `/api/auth/me`, so `currentUser.role` and `currentUser.name` are undefined after a refresh. Fix: `currentUser = (await res.json()).user`.

## JWT secret and auth token lifecycle

The JWT secret in `src/auth.js` defaults to a hardcoded string:

```js
const JWT_SECRET = process.env.JWT_SECRET || "dev-secret-change-me";
```

In a deployment where `JWT_SECRET` isn't set, every token is signed with a public string, meaning anyone can forge admin tokens. (confirmed)

The login endpoint sets an httpOnly cookie *and* returns the token in the JSON body. Both frontends store that token in `localStorage` and use it for all subsequent requests (Authorization header and Socket.IO handshake auth). The httpOnly cookie is never read by the frontend code — it's purely redundant. Since the apps inject user-provided data into innerHTML (order addresses, customer names, driver names), any XSS payload could exfiltrate the localStorage token. The cookie-based path would mitigate this if the frontends relied on it exclusively. (confirmed)

## Order access control gap

`GET /api/orders/:id` applies only `requireAuth` — any driver can fetch the full details of any order by ID:

```js
router.get("/:id", requireAuth, (req, res) => {
  const order = db.prepare("SELECT * FROM orders WHERE id = ?").get(req.params.id);
  if (!order) return res.status(404).json({ error: "Orden no encontrada" });
  res.json(order);
});
```

The status-update endpoint checks ownership for drivers (`order.driver_id !== req.user.id`), but the read endpoint doesn't. A driver can read customer names, phone numbers, and addresses of orders assigned to other drivers by iterating IDs. (confirmed)

## Order code generation under concurrency

```js
function generateCode() {
  const ts = Date.now().toString(36).toUpperCase().slice(-4);
  return `ORD-${ts}${Math.random().toString(36).slice(2, 4).toUpperCase()}`;
}
```

Two requests within the same millisecond share the timestamp prefix. The random suffix is 2 base-36 characters — 1,296 possible values. The `code` column has a UNIQUE constraint, so a collision throws an unhandled SQLite constraint error (no try/catch around the insert), returning a raw 500 to the client. (likely)

## Socket.IO disconnect and offline detection

When a driver's socket disconnects for any reason, the server immediately writes `status = 'offline'` and emits `driver:offline`:

```js
socket.on("disconnect", () => {
  if (user.role === "driver") {
    db.prepare("UPDATE drivers SET status = 'offline' ...").run(user.id);
    io.to("admins").emit("driver:offline", { id: user.id });
  }
});
```

There's also an interval-based detector that marks drivers stale after 30 seconds. The immediate disconnect handler means a driver in a tunnel for 3 seconds gets marked offline and removed from the admin map. On reconnect, the driver would need to re-toggle online and re-share location. A grace period (10-15 seconds) before the offline write would prevent flickering without hurting eventual consistency. (likely)

## PUT update semantics with falsy values

The `PUT /api/orders/:id` handler uses `value || null` for most fields, which means passing `""` (intentional clear) results in NULL, causing COALESCE to keep the old value. Fields like `items`, `notes`, `customer_phone` cannot be cleared once set. This is a data-integrity asymmetry: you can set a value but never unset it through the API. (confirmed)

## Frontend innerHTML and XSS

Both `dispatcher.js` and `driver.js` render API data into `innerHTML` without sanitization:

```js
card.innerHTML = `
  <div class="order-customer">${order.customer_name}</div>
  ...
`;
```

An admin creating an order with `customer_name: "<img src=x onerror=...>"` propagates that payload to every connected admin and the assigned driver. Combined with the localStorage token storage, this chain escalates to full token theft across all open sessions. (confirmed)

## Broken /api/auth/me response unwrapping

The `/api/auth/me` endpoint returns `{ user: { id, name, email, role } }`. Both frontends assign the full response to `currentUser`:

```js
// driver.js line 81
currentUser = await res.json();
if (currentUser.role !== 'driver') {  // undefined !== 'driver' → always true
  showLogin();
  return;
}
```

In `driver.js`, this means every page refresh sends the user back to the login screen even with a valid token. In `dispatcher.js`, the same pattern means `userName.textContent = currentUser.name` renders "undefined" in the navbar. The fix is `currentUser = (await res.json()).user`. (confirmed)
